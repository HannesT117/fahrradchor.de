# Commit Message Convention

## Structure

```
<type>(<scope>): <subject>

<body>

<footer>
```

## Type
Must be one of:
- **feat**: A new feature
- **fix**: A bug fix
- **docs**: Documentation only changes
- **style**: Changes that don't affect code meaning (formatting, whitespace)
- **refactor**: Code change that neither fixes a bug nor adds a feature
- **perf**: Performance improvement
- **test**: Adding or updating tests
- **chore**: Changes to build process or auxiliary tools
- **ci**: Changes to CI configuration

## Scope
The scope should indicate the area of the codebase:
- `intern` - Internal voting system
- `konzerte` - Concert pages
- `landing` - Landing page
- `components` - Shared components
- `ai` - AI/automation related
- etc.

## Subject
- Use imperative mood ("add" not "added" or "adds")
- Don't capitalize first letter
- No period at the end
- Max 50 characters

## Body (optional)
- Explain what and why, not how
- Use bullet points with `-` for multiple items
- Only include the most important user-facing or architectural changes
- Avoid implementation details (e.g., "add interfaces" is an implementation detail)
- Wrap at 72 characters

## Footer
- **Required**: `Committed by Claude Code` (for AI-generated commits)
- Reference issues: `Closes #123` or `Relates to #456`
- Breaking changes: `BREAKING CHANGE: description`

## Examples

### Simple commit
```
fix(konzerte): correct date formatting in concert list
```

### Commit with body
```
refactor(intern): add type safety to voting system

- Replace generic download() with typed wrappers
- Improve type coercion for better readability

Committed by Claude Code
```

### AI-generated Commit 
```
feat(landing): add newsletter signup form

- Add EmailCapture component
- Integrate with Mailchimp API
- Add privacy notice

Committed by Claude Code

Closes #42
```
